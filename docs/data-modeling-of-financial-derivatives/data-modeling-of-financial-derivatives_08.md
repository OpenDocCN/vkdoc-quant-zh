# 本书采用了哈利·埃利斯和理查德·巴克开发的`计算机辅助软件工程`（CASE）方法论以及`实体关系图`（ERD）建模记号法。¹ 巴克在加入甲骨文公司后，进一步发展和扩展了这套 CASE 方法论，并将其作为 Oracle 定制开发方法（CDM）推向市场。时至今日，Oracle 的 CASE 工具仍能使用户通过巴克的记号法进行建模。² 我运用巴克的建模记号法和 Oracle SQL Developer Data Modeler（3.1.4 版）——以下简称`Oracle Data Modeler`——开发了本书中的每一个数据模型。

在本书的撰写过程中，我限定自己使用`Oracle Data Modeler`，并非为了展示某一特定的 CASE 产品，而是因为我发现它是创建易于理解、无歧义、直观且用户友好的 ERD 图的最佳工具。本书呈现的每一个模型都是概念层模型。这一需求要求数据建模产品具备一个预置的、行业公认的概念层数据建模结构库，以便读者能够轻松识别和理解。`Oracle Data Modeler`恰好拥有一个开箱即用的、包含多种概念建模结构的预置库，完美契合本书的标准。例如，该产品展示超类型和子类型实体的方式就非常直观且高度详细。我需要的另一个特性——互斥弧——也随时可用，并具有很高的易用性。

同样，我之所以采用巴克的记号法，是因为它直观且易于掌握。在我的职业生涯中，我得出结论：使用这种特定记号法系统创建的 ERD 图，对于非技术用户来说是最容易解读的。所谓`巴克的记号法`，我不仅指记号法本身，也包括巴克在空间上组织数据模型的方法（巴克的`位置约定`）。当你将这些要素结合在一起时，得到的图表将简洁明了、易于阅读、解读和理解——这正是它们在金融领域发挥有效作用所必需的。因此，第 2 章专门介绍了巴克的 ERD 记号法和方法论。如果你已经熟悉巴克的方法论，可以快速浏览第 2 章。即便如此，我仍建议你至少简要回顾一下这一章，因为它是重温知识的好地方。

第 3 章向读者介绍了金融合约的世界。充分理解这一章很重要，因为它介绍了贯穿全书的关键概念和基本建模模式。

第 4 章至第 8 章讨论了各种金融工具，并指导你如何为它们建模。每章都配有清晰的建模图，以直观易懂的方式解释每种工具。第 9 章则为贯穿本书的一些关键思想和主题画上了点睛之笔。各章内容循序渐进，应按顺序阅读。

## 数据建模模式的益处

识别和应用建模模式的能力来自实践和经验。在数据建模模式的第一本书中，大卫·海伊这样描述它们的重要性³：

> *如果你要追求对业务至关重要的事物，你会发现所有企业都有共通之处——人员、组织、产品、合同在商业世界中是相当标准的。如果你以此作为组织的基础，你最终得到的模型将足够具体，以至于人们能够识别并理解它。*

一个成功的数据模型，既能回答今天提出的具体业务问题，又足够通用，能够预测明天可能发生的各种变化。数据建模模式使你能够创建易于扩展和修改的稳健设计。掌握建模模式能带来更通用、更稳定的数据建模设计，一个能够预见变化的设计。在数据建模中预见变化，通常能产生使用寿命更长、所需侵入性改动更少的设计。如果你从事数据建模工作，你很可能清楚，向分类表中添加一行远比在系统生产部署后才发现需要一个新超类型并需要进行大规模重编码过程要容易得多。

如果你在构建模型时始终以最终客户而非技术为中心，你会注意到最终模型会有多么显著的不同。这种方法的鼻祖威廉·爱德华兹·戴明并非数据建模师，但他的核心原则同样可以应用于数据建模及许多其他学科。

戴明（1900–1993）是一位美国统计学家和商业顾问，以其“计划-执行-检查-行动”循环（即通常所说的戴明环）而闻名。应道格拉斯·麦克阿瑟将军的请求，他在战后的日本教授高层管理人员如何应用统计方法来改进制造业设计、产品质量测试以及在全球市场的销售。戴明准确预测到日本将迅速成为一个向世界市场出口最可靠、最值得信赖的产品和服务的国家。

戴明的一个关键原则是用优质的产品取悦付费客户，并让他们成为回头客。他的原则建立在常识之上，与当今盛行的短视、“即刻见效”的做法背道而驰。戴明主张生产者应从一开始就正确地构建产品，在行动之前以深思熟虑和明智的方式规划和开发产品。一旦质量到位，人们会因为知道你提供的是优质产品而回来再次购买。日本的许多公司坚持戴明的原则并获得了巨大回报。

你如何将戴明的原则应用于数据建模？如果你正确地设计了自己的模型——在必要时进行概括和特化，尽可能依赖数据建模模式和最佳设计原则——那么你的设计将经受住时间的考验，无需进行任何重大的改动，这会让你的客户满意，也让你自己感到欣慰和自豪。

思考以下问题：在一个典型的企业项目中，分配给数据模型开发（包括最终用户访谈和文档编写）的时间通常远少于分配给编码和测试最终系统的时间。无论系统代码多么出色，如果数据库设计糟糕，最终应用程序的运行速度也会很慢。最终用户往往会避开运行缓慢的系统，因此这类系统要么过早退役，要么成为重大改造的对象。不要因为未能分配足够的时间来开发和微调最终的数据模型而犯下这个错误。从长远来看，你将获得巨大的回报。

## 本书中的数据类型

本书提供的数据模型均为*概念级数据模型*，所有超类型和子类型实体均保留原样。这种方法可确保物理级设计能够响应利益相关方批准的业务需求。业务键、备用键、主键的识别以及子类型/超类型的转换，都完全依赖于最终用户的业务规则。若在本著作中包含业务键和备用键等概念，反而会使核心概念复杂化，并可能误导读者。我的意图是尽可能深入地进行数据建模练习，同时为您留下独立探索的空间。请记住，唯有通过不断的实验与实践，我们才能真正学到东西。

将本书的数据模型保持在概念层面的另一个原因，是为了使其与特定数据库管理系统（DBMS）的实现解耦。您的组织为特定项目采用的 DBMS 将拥有众多独特功能，这些功能可能倾向于某种特定的实现方法。作为一名顾问，我曾与众多 DBMS 打交道，它们的独特功能在很大程度上影响了我将概念模型转化为逻辑模型及最终物理模型时所做的决策。这些决策与概念数据模型无关，概念数据模型需要保持中立，不依赖于各种实现方法。尽管我职业生涯的大部分时间都在使用 Oracle DBMS，但我的概念模型始终保持数据库中立。本书也遵循同样的原则。

## 实践与专注

怎样才能进入卡内基音乐厅？唯有通过实践与专注。要精通数据建模亦是如此。要掌握新的数据建模技术，你需要刻苦练习、反复实践。在生活中，你可能会遇到一些看似对数据建模了如指掌，但却鲜有实际技能展示的从业者。他们无意中成了被动的观察者或乘客，只是对他们所见的事物进行建模，没有运用任何创造力和想象力。被动建模确实有效，但其局限性很大，因为它会导致模型僵化，难以应对变化。相反，创造性数据建模则能产生易于修改的模型，因为设计者在构建时就考虑到了变化。通过提问、并且勇于提问来实践创造性建模。通过提问，你能更深入地理解主题领域，并且相比被动建模者，你凭借新鲜的视角和无偏见的观点而占据优势。

实践之所以加倍重要，是因为它能为你提供业务利益相关方的反馈，并鼓励你尝试不同方法。缺乏实践，你的技能就会衰退甚至消亡。通过实践，你将能够将建模技能提升到更具创造性和创新性的水平。单凭知识和实践本身并不足以产生最佳成果。缺乏热情的实践只会带来平庸乏味的结果。要在你所做的事情上做到最好，你需要对工作充满热情。热情将驱使你成为所在领域的佼佼者，并始终以最佳状态投入工作。

## 提出正确的问题

大多数项目都包含一些不熟悉的元素。通常，项目的这一小部分会与截止日期和其他压力相关联。有些人能够迎接挑战，而另一些人则表现不佳。在这种情况下，你需要认识到，如果没有对底层业务规则的正确理解，你就无法交付高质量的解决方案。你的项目中总会有人能够回答你的问题。通常情况下，作为数据建模师，你的任务就是找到这些领域专家，与他们面谈，并挖掘出尽可能多的信息。从人那里获取信息是一门艺术，你应该尽可能多地实践。

首先，向领域专家承认你在某个特定主题领域的知识有限，这是完全可以接受的；这种坦诚通常能缓和对方的情绪，使他们更同情你的需求，感觉到自己的重要性，从而更健谈。

当你与领域专家访谈时，要尽量减少会导致简单“是”或“否”答案的*封闭式问题*，因为它们会阻碍富有成效的对话交流。相反，应该提出旨在鼓励充分披露和有意义讨论的*开放式问题*。例如，请领域专家解释一个特定的业务流程，将有助于促成高质量且富有成效的讨论。

## 结论

本章讨论了各种主题，从本书的整体结构到识别和应用各种建模模式的重要性。它强调了：如果希望在行业中取得成功，学习的渴望和获取新信息的意愿都是关键品质，被动接收信息是不够的。只有通过反复的实际知识应用，我们才能学到东西并取得成功。当涉及到提问时，要无所畏惧，这样我们才能掌握任何特定的业务领域。最后，要么满怀热情地去做，要么就干脆不做！
